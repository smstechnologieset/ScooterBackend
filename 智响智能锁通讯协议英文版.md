## 1 Smart Lock Basic Function

## 2 Smart Lock Identification

This Identification is the lock number actually, showed in the name of BT(blue tooth lock) Broadcast. For example, in the BT broadcast "GK:xxxxxxxxxxxVVV", GK letters is not changeable,"-" show it broadcasts common.
When "-" changes into ":", it shows press the key on the lock. Among the broadcast,12 digits"xxxxxxxxxxx" means the lock number, and VVV shows the lock battery voltage.

## 3 Protocol Format

### 1. BT&Phone APP

The lock and App communicate to each other by Data Package. Once the BT connection is made by the phone, if the connection hold time is over 10 seconds with bike still locked, the lock will give up connection. And BT connection works after the lock turn to open, it won't get disconnected until the phone APP's disconnection requirement or the lock is closed and successfully uploads feedback.

BT transmission content can't be over 20 bytes for one time, Otherwise it should be divided into parts.

Service UUID:0x6E400000B5A3F393E0A9E50E24DCCA9E

Feature：0002 written by APP

Feature: 0003 sent by BT

**Data Package Format**

| Starting Section | Message Length Section | Message Function Section | Message Section | Check Section |
|---|---|---|---|---|
| 2bytes（0x7B,0x5B） | 1byte | 1byte | n-byte（max 255） | 1byte |

1. Message Length Section means the length of Bytes of Message section.
2. Message Function Section means the functions like unlock.
3. Message Section means the detail information shown in this part.
4. Check Section is the xor of all bytes on message function section and message section.

## 4 Function Details

### 4.1 Check the bike and get random number

This function will receive random code to open the lock for encryption.

**1. App send：**

| Starting Sec. | Length Sec. | Function Sec. | Parameter | Check Sec. |
|---|---|---|---|---|
| 0x7B5B | 0x01 | 0x61 | 1byte | 1byte |

For example: 0x7B5B0161 00 61

Parameter: 00 shows only check the bike status; 01 shows check the bike status, and the speaker rings.

**2. Reply by BT:**

| Starting Sec. | Length Sec. | Function Sec. | Error Code | Hardware version | Software version | In rental | Random code | Battery voltage (hex） | Check Sec. |
|---|---|---|---|---|---|---|---|---|---|
| 0x7B5B | 1byte | 0x91 | 1byte | 1byte | 1byte | 1byte | 4bytes | 2bytes | 1byte |

For random code 4B, it is a integer data (big-endian), needs to convert it into 8B bytes for unlock encryption. Like received 0x1f3e332a, needs to turn it into characters "1f3e332a", note the letters in lower case, and save for the next step that open the lock by BT.

In rental: 0x00-ready 0x01-in rental.

Error code: ignore.

Battery voltage: the value is little-endian, such as 0x9001,convert to 0x0190,then the battery voltage is 4.00V.

### 4.2 Unlock

**App send：**

| Starting Sec. | Length Sec. | Function Sec. | Key Index | User Id | Time Stamp | AES(rand,key) encryption | Check Sec. |
|---|---|---|---|---|---|---|---|
| 0x7B5B | 1byte | 0x62 | 1byte | 6bytes | 4bytes | 16bytes | 1byte |

**Reply by BT:**

| Starting Sec. | Length Sec. | Function Sec. | Error Code | Check Sec. |
|---|---|---|---|---|
| 0x7B5B | 1byte | 0x92 | 1byte | 1byte |

**Encryption :**

Every lock will get 12-characters lock identification number after production, and that lock number corresponds to 80-characters key number.

Such as

| QR Code No. | Lock number | Key number |
|---|---|---|
| 100002 | CED22BB62C02 | Xa8ynGBS4GdJgN7JtkK56kvXRDbXzBCpMBHgrF3gLKQUlK2RwyoRWC0GVCerIJ1rhQTkPQiwOXAwm0YB |

The lock number and key number, will be recorded in the servers system. When user scans the two dimensional code(QR Code), App access the server, and get the MAC(lock number) and 80 characters Key Number. The key index is generated by the phone APP or Sever randomly, the value range is 0-64. Such as the index is 0, then you will get 16-characters key "Xa8ynGBS4GdJgN7J" from above table. And such as the rand numbers from "get random number" function are "1f3e332a", then fill in 8-chareacters '0' closely followed. So 16-characters rand=" 1f3e332a00000000".

Then calculate AES(rand,key)- ECB, its hexadecimal data(16bytes) is 0x71659CB81453F7DD68B21EC5FD2C10BC.

**Get Open Lock Data:**

So if the 6 bytes User ID is 0x013686020001, the 4 bytes Time Stamp (Greenwich Time) is 0x 58920600,

The data to open lock is 0x7B5B 1B 62 00 013686020001 58920600 71659CB81453F7DD68B21EC5FD2C10BC 3D.

the check section is 0x3d from xor(62 00 013686020001 58920600 71659CB81453F7DD68B21EC5FD2C10BC) In this example.

**Send to BT:**

Sent to BT lock data in divided parts(every part is less than 20 bytes). If no feedback after 10 seconds, you can confirm the lock is open or not by the "get the record" function.

**Reply Error Code by BT:**

00: success
FF: unknown mistake
FE: battery capacity is extremely low
FD: Wrong Check section or Wrong Encryption
FC: Deal record is full
FA: Unlocking failed, mechanical breakdown
F9: The time of receiving unlocking order is overtime
F6: Unlock failure, mechanical failure, it doesn't reach the unlock position.
F6: Unlock failure, mechanical failure, it doesn't leave the lock position.

### 4.3 Get the Record

After lock opens, there is a record, containing mark (user ID and unlocking timestamp), situation and unlocking time. Same case with that lock closes, a record contains mark, situation and locking time. This record is saved by a first-in first-out way.

Before unlock , after lock or when check lock status, App get the record by below steps.

App send "read record" function first, then wait for reply by BT. If have a valid record, App need confirm record received , send it back for BT to clear this record, then the BT will reply the next record. If have a "null record", then it is end of "get the record" .

**App send "read record"：**(0x7b5b006868)

| Starting Sec. | Length Sec. | Function Sec. | Check Sec. |
|---|---|---|---|
| 0x7B5B | 1byte | 0x68 | 1byte |

**Reply "read record" by BT:**

A valid record

| Starting Sec. | Length Sec. | Function Sec. | Current lock status | Lock number | Open Time Stamp | User ID | Record type | Record Time Stamp | AES Encryption | Key Index | Check Sec. |
|---|---|---|---|---|---|---|---|---|---|---|---|
| 0x7B5B | 1byte | 0x98 | 1byte | 12bytes | 4bytes | 6bytes | 1byte | 4byte | 16bytes | 1byte | 1byte |

A null record

| Starting Sec. | Length Sec. | Function Sec. | Current lock status | Check Sec. |
|---|---|---|---|---|
| 0x7B5B | 1byte | 0x98 | 1byte | 1byte |

**Confirm "record read" by APP to send back:** ( Note: no need current lock status)

| Starting Sec. | Length Sec. | Function Sec. | Lock number | Open Time Stamp | User ID | Record type | Check Sec. |
|---|---|---|---|---|---|---|---|
| 0x7B5B | 1byte | 0x68 | 12bytes | 4bytes | 6bytes | 1byte | 1byte |

Open Time Stamp: time stamp same as "unlock" function .

Current lock status: 0x00 –unlock status 0x01- lock status;

Record type: 0x00- unlocked record 0x01- locked record;

Record Time Stamp : unlocking time or locking time.( Greenwich);

AES Encryption &Key Index: only use for Specific user, so this section ignores.

### 4.4 Unlocking BT long connection

After unlocking, APP will take the BT connection as verified. When disconnection happens during the riding, APP should request connection again, and send verification code. if not, the lock will get disconnected automatically after 10 seconds.

**Send verification code：**

| Starting Sec. | Length Sec. | Function Sec. | User ID | Check Sec. |
|---|---|---|---|---|
| 0x7B5B | 1byte | 0x66 | 6bytes | 1byte |

**Reply by BT:**

| Starting Sec. | Length Sec. | Function Sec. | Error code | Check Sec. |
|---|---|---|---|---|
| 0x7B5B | 1byte | 0x96 | 1byte | 1byte |

Error code: 0x00- certificated 0x01-uncertificated

User ID: same as "unlock" function.

### 4.5 Report locked

After locking by hand, if there is certificated BT connection, APP will receive:0x7b5b009999. Then APP go in to read the record, ends bike using.

**Report by BT:**

| Starting Sec. | Length Sec. | Function Sec. | Check Sec. |
|---|---|---|---|
| 0x7B5B | 1byte | 0x99 | 1byte |
